## 目录职责

### `app/browser`

负责浏览器和登录态相关能力：

- 启动 / 关闭 Playwright
- 复用 `storage_state.json`
- 自动登录
- `reCAPTCHA` 人工补充验证
- 保存更可读的登录态 JSON

### `app/core`

负责项目统一配置：

- Pixiv 账号密码
- 是否无头模式
- 超时时间
- 图片目录
- 数据库路径
- 日志路径
- 日志初始化和输出级别

### `app/crawler`

负责页面采集：

- `artwork_crawler.py`：打开作品详情页、抓取 HTML、保存 HTML / JSON
- `author_crawler.py`：打开作者作品页、关注列表页，提取作者名下作品 ID

### `app/parser`

负责把作品页 HTML 解析成结构化数据：

- 标题
- 作者
- 标签
- 页数
- 图片候选地址
- 当前页真实作品 ID
- `__NEXT_DATA__` 调试命中信息

### `app/downloader`

负责图片下载：

- `download_planner.py`：生成下载计划，补页码、补候选 URL
- `download_path_builder.py`：负责目录和文件名规则
- `image_downloader.py`：复用浏览器 cookies 和代理执行实际下载
- 自动选原图
- 多图作品自动全下
- 自动判断是否已下载
- 本地文件缺失时支持自愈重跑
- 图片按“一个作者一个文件夹”存放
- 文件名按“作品标题 + 作品 ID”生成

### `app/db`

负责 SQLite 任务记录：

- 保存处理状态
- 查询历史
- 筛选失败记录
- 统计状态和错误类型
- 删除旧记录
- 失败重试时保留历史元数据

### `app/services`

负责主流程辅助逻辑：

- `cli_service.py`：终端菜单、输入读取、历史查看、导出、归档
- `console_service.py`：终端展示层，负责摘要、列表、提示信息
- `task_service.py`：单作品处理、批量任务、作者增量筛选、本地文件自愈
- `failure_classifier.py`：失败原因分类
- `failure_exporter.py` / `record_exporter.py`：导出失败清单和归档记录

### `app/utils`

负责调试辅助：

- HTML / JSON 格式化

## 当前图片保存规则

图片现在按下面的方式保存：

- 一个作者一个文件夹
- 文件名优先用作品标题
- 同时带上作品 ID，避免重名覆盖

示例：

```text
data/images/mignon_24234/引きこもりの妹が制服を着てみた回__142543623.jpg
data/images/mignon_24234/制服まとめ__123456789_p0.jpg
data/images/mignon_24234/制服まとめ__123456789_p1.jpg
```

## `data` 目录现在怎么理解

### `data/images`

正式下载结果目录。  
这是运行产物，不建议上传 GitHub。

### `data/state`

浏览器登录态目录。  
里面的 `storage_state.json` 含有登录 cookies，属于敏感文件，绝对不要上传 GitHub。

### `data/exports`

导出的失败清单和归档文件目录。  
属于运行产物，不建议上传 GitHub。

### `data/temp`

本地临时调试目录。  
主要用来存最近抓下来的 HTML / JSON 方便排查问题。  
当前已经加入 `.gitignore`，不再作为仓库长期样本目录。

### `data/pixiv.db`

SQLite 任务数据库。  
记录每个作品的处理状态、标题、作者、下载数量、错误信息等。  
属于运行产物，不建议上传 GitHub。

## Docker 相关文件

当前仓库已经支持 Docker：

- `Dockerfile`
- `docker-compose.yml`
- `.dockerignore`

推荐用法：

1. 先在宿主机跑通一次登录
2. 生成 `data/state/storage_state.json`
3. 再让容器复用这个登录态运行

原因是 Pixiv 登录可能会触发 `reCAPTCHA`，而容器默认更适合无头运行。
