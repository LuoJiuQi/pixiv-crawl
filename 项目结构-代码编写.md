# 项目结构 - 代码编写

这份文档讲的是：  
**现在这项目的代码是怎么分层的，后面继续写功能应该往哪里放。**

---

## 一、现在主流程是怎么串起来的

程序入口在：

```text
main.py
```

你运行：

```powershell
python main.py
```

以后，主流程大致是这样：

1. 读取配置
2. 初始化数据库
3. 让用户在终端里选模式
4. 如果需要访问 Pixiv，就启动浏览器
5. 检查并复用登录态
6. 根据模式收集作品 ID
7. 把作品 ID 交给批量任务处理
8. 保存数据库记录、下载图片、输出结果

---

## 二、代码分层怎么理解

### `main.py`

这里只做“总调度”：

- 选功能模式
- 组装浏览器、数据库、爬虫、下载器
- 调用服务层

这里尽量不要继续堆太多业务细节。  
如果某段逻辑开始变长，就应该往 `services/` 里拆。

---

### `app/browser`

这一层只放“浏览器和登录”相关代码。

当前分工：

- `client.py`
  - 启动浏览器
  - 创建 context / page
  - 保存登录态
  - 关闭资源

- `login.py`
  - 自动登录
  - 检测是否已登录
  - `reCAPTCHA` 人工处理

- `state_manager.py`
  - 管理 `storage_state.json`
  - 判断是否存在
  - 删除失效登录态
  - 用可读格式保存状态文件

这一层不要写：

- 作品解析
- 图片下载
- 数据库逻辑

---

### `app/crawler`

这一层只负责“把网页内容拿下来”。

当前分工：

- `artwork_crawler.py`
  - 打开作品详情页
  - 获取 HTML
  - 保存 HTML / JSON 快照

- `author_crawler.py`
  - 打开作者页
  - 从作者页拿作品 ID
  - 支持接口优先、DOM 兜底

这一层不要写：

- HTML 字段解析
- 数据库存储
- 图片命名规则

---

### `app/parser`

这一层只负责“解析”。

当前核心文件：

- `artwork_parser.py`

负责从 HTML 里提取：

- 标题
- 作者
- 标签
- 页数
- 图片候选地址
- `__NEXT_DATA__` 命中信息

如果后面你要加：

- 作者资料解析
- 标签页解析
- 排行榜条目解析

也应该优先放到 `parser/`。

---

### `app/downloader`

这一层只负责“下载图片”。

当前核心文件：

- `image_downloader.py`

它现在负责：

- 根据候选 URL 生成下载计划
- 自动补多图页
- 利用浏览器 cookies 下载图片
- 判断本地是否已下载
- 生成最终图片路径

当前图片保存规则是：

- 一个作者一个文件夹
- 文件名：`作品标题__作品ID`
- 多图：`作品标题__作品ID_p0`

如果以后你要改：

- 命名规则
- 目录结构
- 下载重试
- 图片校验

优先改这里。

---

### `app/db`

这一层只负责 SQLite。

当前核心文件：

- `download_record_repository.py`

它负责：

- 建表
- 插入 / 更新记录
- 查记录
- 统计状态
- 删除旧记录

这一层不要写：

- 终端交互
- HTML 解析
- 浏览器操作

---

### `app/services`

这一层是“业务逻辑层”。

当前分工：

- `cli_service.py`
  - 菜单选择
  - 输入解析
  - 历史查看
  - 导出和归档

- `task_service.py`
  - 单作品处理
  - 批量处理
  - 作者增量筛选
  - 汇总输出

- `failure_classifier.py`
  - 错误分类

- `failure_exporter.py`
  - 失败记录导出

- `record_exporter.py`
  - 通用记录导出

如果你以后继续扩功能，通常最优先考虑往这里加。

例如：

- 作者订阅名单
- 标签任务
- 排行榜任务
- 定时同步任务

---

### `app/schemas`

这一层放“数据长什么样”的标准定义。

当前：

- `artwork.py`

这里的作用是让别的模块统一使用同一种作品数据结构，  
避免到处乱拼字典。

---

### `tests`

这一层放回归测试。

当前已经覆盖了：

- 主流程输入解析
- 解析器
- 下载器
- 数据库
- 失败分类
- 失败导出
- 记录导出
- 作者抓取
- 作者增量筛选

简单理解：

- 功能写完后
- 最好顺手补对应测试

## 三、如果后面继续加功能，建议怎么放

### 1. 按作者订阅更新

建议放：

- `app/services/`
- `app/db/`

因为它本质上是任务管理，不是页面解析。

### 2. 标签抓取 / 排行榜抓取

建议放：

- 页面进入逻辑放 `app/crawler/`
- 列表项解析放 `app/parser/`
- 批量任务编排放 `app/services/`

### 3. 图片命名和目录规则

建议放：

- `app/downloader/image_downloader.py`

### 4. 终端菜单新选项

建议放：

- `app/services/cli_service.py`
- `main.py`

---

## 四、现在写代码时最重要的原则

### 原则 1

`main.py` 不要再变成大杂烩。  
能拆到 `services/` 的，尽量拆。

### 原则 2

“拿网页内容”和“解析网页内容”要分开。  
也就是：

- `crawler` 负责拿
- `parser` 负责拆

### 原则 3

“下载逻辑”和“任务调度”分开。  
下载器不应该负责问用户选项，  
调度层也不应该直接管图片文件怎么命名。

### 原则 4

运行产物不要混进仓库。  
比如：

- `data/images`
- `data/state`
- `data/temp`
- `data/pixiv.db`

这些都应该只留在本地。

---

## 五、现在读代码最推荐的顺序

如果你以后想自己回来看，推荐按这个顺序读：

1. `main.py`
2. `app/services/cli_service.py`
3. `app/services/task_service.py`
4. `app/browser/client.py`
5. `app/browser/login.py`
6. `app/crawler/artwork_crawler.py`
7. `app/crawler/author_crawler.py`
8. `app/parser/artwork_parser.py`
9. `app/downloader/image_downloader.py`
10. `app/db/download_record_repository.py`

这个顺序最符合“程序真正是怎么跑起来的”。
